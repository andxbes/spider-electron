# Підтримка документації Spider-Electron

Інструкція для розробників і AI-агентів: коли і як оновлювати внутрішню документацію після змін у коді.

## Файли документації

| Файл | Призначення |
|------|-------------|
| [ARCHITECTURE.md](./ARCHITECTURE.md) | Архітектура, IPC, моделі даних, константи, команди |
| [DOC_MAINTENANCE.md](./DOC_MAINTENANCE.md) | Цей файл — правила оновлення |
| `.cursor/rules/spider-electron.mdc` | Cursor rule: нагадування оновлювати docs при правках |

## Коли оновлювати ARCHITECTURE.md

Оновлення **обов'язкове** в тому ж PR/коміті, що й код, якщо змінилось хоча б одне з нижче:

### Main / краулер (`src/main/main.js`)

- [ ] `MAX_PAGES_TO_VISIT`, timeout, User-Agent, правила домену
- [ ] Алгоритм обходу (BFS → інше), черга, ліміти
- [ ] Обробка robots.txt, redirects, nofollow
- [ ] Поля в об'єкті `spider-result`
- [ ] Нові/видалені IPC-канали

### Preload (`src/preload/preload.js`)

- [ ] `validSendChannels` / `validReceiveChannels`
- [ ] Методи `window.api`

### Renderer (`src/renderer/`)

- [ ] Нові UI-елементи або потоки даних
- [ ] Колонки / формат CSV export
- [ ] Зміни в `scanResults` або відображенні

### Інфраструктура

- [ ] `package.json` scripts або залежності (таблиця deps)
- [ ] Нова структура каталогів / entry points
- [ ] `forge.config.js`, build pipeline, CSP

### Не потребує оновлення ARCHITECTURE.md

- Косметичні зміни UI (колір, відступи) без зміни поведінки
- Рефакторинг без зміни публічної поведінки (але варто перевірити, чи номери рядків у docs застаріли)
- Коментарі в коді

## Чеклист після правки

1. Прочитати [ARCHITECTURE.md](./ARCHITECTURE.md) і знайти зачеплені секції.
2. Оновити відповідні таблиці, діаграми, константи.
3. Змінити рядок **«Останнє оновлення»** на початку ARCHITECTURE.md на поточну дату.
4. Якщо додано новий тип змін, яких немає в цьому чеклисті — дописати пункт у DOC_MAINTENANCE.md.

## Шаблон запису в ARCHITECTURE.md

При додаванні нової можливості:

```markdown
### [Назва можливості]

- **Файл:** `src/...`
- **Тригер:** як запускається
- **Поведінка:** 1–3 речення
- **IPC / дані:** якщо є
```

## Для AI-агентів (Cursor)

Правило `.cursor/rules/spider-electron.mdc` з `alwaysApply: true` нагадує:

- Перед правками — прочитати `docs/ARCHITECTURE.md`.
- Після правок, що змінюють поведінку — оновити docs у тому ж сеансі.
- Не створювати дублікати (README vs ARCHITECTURE) без запиту користувача.

## Майбутнє розширення

Якщо проєкт виросте, розділити docs так:

```
docs/
├── ARCHITECTURE.md    # загальна архітектура (залишити)
├── IPC.md             # детальний контракт каналів
└── CHANGELOG.md       # для користувачів (за потреби)
```

Поки проєкт малий — один ARCHITECTURE.md достатньо.
